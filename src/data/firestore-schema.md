# Firestore Schema for Cognitive Crucible

This document outlines the proposed Firestore database schema for storing user performance data, designed for scalability and mode-independent analysis.

## 1. Unified Tracking & Schema (`users` collection)

To track user performance independently of the game "Mode" (or "Focus") while still allowing for mode-specific queries, we use a flat collection for trial data and a separate collection for the adaptive game state.

**Collection:** `users`
**Document ID:** `{userId}`

---

### 1.1. User Game State (`game_states` sub-collection)

This collection stores the adaptive state for each game *and* focus combination. This ensures that a user's progress in "Gwm / Music" is tracked separately from "Gwm / Math".

**Sub-collection:** `game_states`
**Document ID:** `{gameId}/{focus}` (e.g., "gwm_dynamic_sequence/music")
**Document Fields:** This document stores the `AdaptiveState` object as defined in `src/types/index.ts`.

```json
{
  "gameId": "gwm_dynamic_sequence",
  "lastFocus": "music",
  "currentLevel": 5,
  "levelFloor": 1,
  "levelCeiling": 10,
  // ... other AdaptiveState fields
}
```

### 1.2. User Trial Logs (`events` sub-collection)

This collection stores a record for every single event that occurs, including trials, session starts, and errors. Storing events in a single, flat collection is crucial for efficient querying and analysis.

**Sub-collection:** `events`
**Document ID:** Auto-generated by Firestore (`doc()`)

**Document Fields:** The structure mirrors the `TelemetryEvent` from `src/lib/telemetry-events.ts`.

```json
{
  "eventId": "evt-1677200000-123",
  "sessionId": "session-abc-123",
  "timestamp": 1677200000123,
  "schemaVersion": 2,
  "seq": 15,
  "type": "trial_complete",
  "payload": {
    "gameId": "gwm_dynamic_sequence",
    "focus": "music", // The mode played for this trial
    "difficultyLevel": 5,
    "correct": true,
    "rtMs": 789,
    "stimulusParams": {
      "melody_length": 4,
      "melody_notes": [60, 64, 67, 60]
    }
    // ... other payload fields
  }
}
```

**Querying Strategy:**

*   **Overall performance for a game:** `collection('users/{userId}/events').where('payload.gameId', '==', 'gwm_dynamic_sequence').where('type', '==', 'trial_complete')`
*   **Performance for a specific mode:** `collection('users/{userId}/events').where('payload.gameId', '==', 'gwm_dynamic_sequence').where('payload.focus', '==', 'music').where('type', '==', 'trial_complete')`
*   **Performance over time:** `...orderBy('timestamp', 'desc').limit(100)`

This structure ensures that core metrics (`correct`, `rtMs`) are consistently present, while mode-specific details are contained within the `stimulusParams` object, satisfying the decoupling requirement.

## 2. Stimulus & Asset Management

This schema intentionally **avoids** storing large assets (audio files, 3D models, large word lists) directly in Firestore/RTDB.

*   **Verbal/Music Content:** For small, static datasets like word lists or music theory questions, these are best bundled directly with the client application in the `/src/data/` directory (as is current practice). This provides the lowest possible latency.
*   **Larger Assets (Audio, Spatial Models):** These should be stored in **Firebase Storage**. The client application should fetch these assets on demand and cache them aggressively using a **Service Worker** and/or **IndexedDB**. Firestore documents should only store *references* to these assets (e.g., the Firebase Storage path `gs://my-app.appspot.com/assets/music/C4.mp3`), not the assets themselves.
